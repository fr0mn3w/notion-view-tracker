# PRD: Weekly Report (Social Metrics to Notion)

## Goal
A Notion "Weekly Report" database that publishes one row per week summarizing how the
team's channels performed, views-wise, with an AI-written narrative of the week. It is a
rollup layer on top of the existing Daily Snapshots pipeline, not a new data source.

Phase 1 (this PRD) ships the views numbers and the summary. Phase 2 adds the content side
(what was published, what is on the docket) and per-item attribution by linking to the
Media database.

## Locked decisions
- Output: a single new Notion database, "Weekly Report", one row per week.
- Week boundary: Sunday to Saturday. The report for a finished week is generated at the
  start of the following week.
- Granularity: one row per week, with per-channel columns plus one combined summary.
- Weekly views are computed from the data already in Daily Snapshots. The weekly job does
  NOT call X or YouTube. Zero added platform API cost, no new platform auth.
- X weekly views use a week-over-week delta (X has no clean per-day series).
- YouTube weekly views are the true sum of that week's daily plays.
- The narrative summary is generated by Notion's native AI (an AI autofill property on the
  database), not by our pipeline. Our weekly job writes only the numbers and stays fully
  deterministic. No LLM runs in our code, and no Anthropic API key is needed.

## Deferred to phase 2 (deliberately)
- "What was published" this week (needs the Media database).
- "What is on the docket" (in production / editing / to be reviewed): production status
  that lives in the Media database.
- "What generated the views": per-item attribution (which post or video drove the week's
  views). Requires collecting top-performing items per platform and relating them to Media.
- Notion relations / rollups between Weekly Report and the Media database.

## Architecture
A second scheduled GitHub Actions workflow, weekly. Reuses the existing repo, secrets, and
Notion integration.

1. Trigger: a weekly cron (e.g. Mondays ~09:00 UTC), summarizing the prior Sunday-Saturday.
   Plus manual dispatch.
2. Read: query the existing Daily Snapshots database for the target week's rows, plus the
   prior week's boundary rows (needed for the X delta and follower deltas). No platform calls.
3. Compute: per-channel weekly views and net follower/subscriber change.
4. Write: upsert one row into the Weekly Report database, keyed by the week, with the
   computed numbers.
5. Summarize: handled inside Notion by an AI autofill property that reads the row's numeric
   columns and writes the narrative. Our job does not call any LLM.
6. Fail soft: if a fetch or compute step errors, write what we can and log it. The summary
   is independent (Notion-side), so it never blocks the numbers.

## Notion schema (single database: Weekly Report)
One row per week. Phase-1 columns:
- Name (Title): the week label and unique upsert key, e.g. "Week of 2026-06-22".
- Week Start (Date): the Sunday.
- Week End (Date): the Saturday.
- Views: newsystems_ (Number): X weekly views (delta, approximate).
- Views: otherstuff X (Number): X weekly views (delta, approximate).
- Views: otherstuff YT (Number): YouTube weekly plays (summed, accurate).
- Total Views (Number): sum of the three.
- Net Followers: newsystems_ (Number): followers gained over the week.
- Net Followers: otherstuff X (Number).
- Net Subscribers: otherstuff YT (Number).
- Summary (AI autofill): generated by Notion AI from the numeric columns (not written by
  our pipeline).
- Generated (Date): when the row was produced.

Phase-2 columns (added later, not built now): Published (relation to Media), On Deck
(production status from Media), Top Drivers (per-item attribution).

## Metric definitions
Computed from Daily Snapshots rows in the week, per channel.

- YouTube weekly views: sum of the daily Views rows for Sun-Sat. Clean and accurate
  because YouTube provides a real per-day time series.
- X weekly views: (Views on the week-end Saturday snapshot) minus (Views on the prior
  Saturday snapshot). Approximate: the X Views column is a rolling 7-day impression total,
  so this is a proxy for weekly change, not an exact weekly count. Flagged as such. We will
  sanity-check it against reality after the first few weeks and may switch to "use the
  Saturday rolling-7-day value directly" if that tracks better.
- Net followers / subscribers: (count on week-end) minus (count on prior week-end). True
  cumulative metrics, so this is exact.

If a boundary snapshot is missing (a skipped day), fall back to the nearest available
snapshot within a small tolerance and note it.

## Summary (Notion AI, not our pipeline)
- Mechanism: an AI autofill property on the Weekly Report database with a custom prompt,
  e.g. "Write a 2 to 4 sentence summary of this week's channel performance from the view
  and follower columns: which channels grew, standout numbers, week-over-week movement."
- It reads the row's numeric columns and writes the narrative inside Notion. Our pipeline
  does not call any LLM and needs no Anthropic API key.
- It cannot discuss content in phase 1 (no published/docket data yet); it speaks only to
  the numbers. Phase 2 enriches the prompt with content context.
- Cost: billed through Notion AI (the team's Notion plan), not per-call.
- Verification needed: confirm the AI autofill regenerates when our pipeline creates or
  updates the row via the Notion API. Notion AI auto-update can be inconsistent on
  API-driven edits. If it does not auto-trigger reliably, the fallback is a plain
  templated summary written by our pipeline (deterministic, no AI).

## Config (added to config.yaml)
- The Weekly Report database id (or NOTION_DATABASE_ID-style env override).
- Week boundary: Sunday-Saturday (fixed for v1, but kept in config).
- The channels to roll up and which Daily Snapshots (platform, account) each maps to.

## Build plan
- Phase 1 (this PRD): weekly workflow + reader + metric math + Claude summary + Notion
  writer for the Weekly Report database. Deterministic numbers, fail-soft summary.
- Phase 2: Media database linking. Pull "what was published" and production status from
  Media; add per-item attribution (top posts/videos) for "what generated the views";
  wire Notion relations/rollups. Gated on understanding the Media database structure.
- Phase 3 (optional): charts/views on the Weekly Report database, longer rollups
  (monthly/quarterly), alerting on big swings.

## What only the team can do
- Create the Weekly Report database with the schema above and share it with the existing
  Notion integration.
- Set up the Notion AI autofill Summary property (custom prompt) on that database. This is
  a Notion-side task, well suited to the Notion AI agent.
- For phase 2: provide access to and the structure of the Media database.

## Open questions for phase 2
- Media database structure: properties, the production-status taxonomy, how a published
  item maps to a channel and date.
- Per-item data collection: do we pull top posts/videos in the daily pipeline or only in
  the weekly job, and how many per platform.
- Whether "what generated the views" is a relation to Media items, a ranked list, or both.
